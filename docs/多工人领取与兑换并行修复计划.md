# 多工人领取、兑换码管理和并行兑换修复计划

## 当前阶段

- 阶段 5：核心实现、评审修复和最终全量验证已完成。

## 已确认目标

- 已使用兑换码不硬删除，改为归档隐藏，保留订单审计关系。
- 工人通过按钮领取任务，领取后订单锁给本人，超时未完成自动释放。
- 多个工人账号按领取和完成归属分开计数。
- 管理统计展示多工人吞吐、任务归属、失败原因和队列并发状态。
- Pix 生成继续使用 BullMQ worker 并发，自动支付检测改为限流并行。
- 客户侧错误继续脱敏，管理员侧展示可排障的安全明文诊断。

## 关键决策

- 不编辑旧 migration，不恢复旧领取死代码，新增当前迁移重新引入必要字段和索引。
- 领取锁默认 30 分钟过期，工人页定时续租当前任务。
- 自动检测完成已领取订单时，计入领取该订单的工人；未领取订单自动完成时，计入自动/未归属。
- 兑换码状态继续从 `usedAt` 派生，不新增兑换码状态枚举。
- 失败诊断只保存阶段、错误码、HTTP 状态和截断后的安全明文，不保存 Session、Pix code、Stripe client secret 或代理密码。

## 实施阶段

### 阶段 1：数据模型和后端边界

- 新增 `redemption_codes.archived_at`。
- 新增 `orders.claimed_by_id`、`orders.claimed_at`、`orders.claim_expires_at`。
- 新增 `orders.generation_error_stage`、`orders.generation_error_detail`、`orders.generation_error_http_status`。
- 恢复工人完成统计和领取查询需要的索引。
- 补充严格查询参数校验。

### 阶段 2：兑换码管理

- 后端支持批次过滤、搜索、批次列表、归档已使用码。
- 未使用码仍支持硬删除，已使用码只允许归档隐藏。
- 前端新增批次过滤、搜索、归档按钮和默认隐藏归档记录。

### 阶段 3：失败诊断

- ChatGPT checkout 和 Stripe Pix 错误携带安全诊断明细。
- Pix 生成失败落库管理员可见诊断字段。
- 客户追踪页继续返回安全中文错误。
- 管理员订单列表展示阶段、错误码、HTTP 状态和诊断明细。

### 阶段 4：工人领取和多工人统计

- 工人接口新增领取下一单、我的任务、续租、释放。
- 完成接口只允许完成本人领取的订单，并写入 `completedById`。
- 工人页改为领取按钮和我的任务列表。
- 工人管理页和 Dashboard 展示工人维度统计。

### 阶段 5：并行检测和验证

- Dashboard 展示 `PIX_WORKER_CONCURRENCY`。
- 自动支付检测使用 `PAYMENT_DETECTION_CONCURRENCY` 限流并行。
- 保留条件更新，避免重复检测导致重复完成。
- 完成后运行 Prisma 校验、后端测试、前端测试、类型检查、构建和 diff 检查。

## 后续注意事项

- 不提交 `graphify-out/`。
- 如果兑换码表数据量很大，后续再评估 `pg_trgm`，本轮只做基础索引和可用搜索。
- 每个阶段结束后更新本文件的当前阶段、已完成内容和验证记录。

## 已完成内容

- 新增兑换码归档字段、订单领取字段、失败诊断字段和统计索引。
- 兑换码管理支持批次过滤、搜索、归档范围筛选、批量归档已使用码和单个已使用码归档。
- 工人任务支持领取下一单、查看我的任务、续租、释放和本人完成。
- 工人完成会写入 `completedById`，自动检测完成已领取订单时计入领取工人。
- 自动支付检测改为按 `PAYMENT_DETECTION_CONCURRENCY` 限流并行。
- 自动检测完成订单时用数据库当前有效领取人原子写入 `completedById`，避免检测期间领取变化导致错记。
- 工人可领取列表只返回未领取或领取已过期订单，不展示其他工人的有效领取任务。
- 管理员订单列表不再返回原始 Pix code，只保留失败诊断和工人归属信息。
- Dashboard、工人管理页、工人页和订单列表均展示多工人相关信息。
- 失败订单保存管理员可见的安全诊断字段，客户侧仍显示安全中文错误。
