# 高并发兑换排队改造计划

## 当前阶段

- 阶段 1：已完成测试边界、Redis + BullMQ 排队下单、生成 worker、代理池、追踪页和后台指标。

## 已确认目标

- 客户提交兑换码后快速返回追踪 token，Pix 生成进入后台队列。
- 并发满时订单保留在队列中，不直接失败。
- 只有维护模式开启时拒绝新订单。
- ChatGPT 和 Stripe 使用独立代理池，Stripe Pix 生成链路固定同一个 Stripe 代理。
- 工人页只显示已经生成 Pix 的 `PENDING_PAYMENT` 订单。

## 关键决策

- `orders.redemption_code_id` 改为可空唯一字段，生成失败时保留失败订单并释放兑换码。
- 生成失败的原始兑换码写入 `submitted_redemption_code`，便于后台审计但不重新占用兑换码。
- 代理健康状态写入系统设置表，API 进程和 worker 进程都能读取同一份失败计数和冷却状态。
- 队列由 Redis + BullMQ 负责，API 只负责创建 `CREATING_PAYMENT` 订单和投递 job。

## 后续注意事项

- 不提交 `graphify-out/`。
- 日志不能输出 Session、Pix code、Stripe client secret 或代理密码。
- 完成每个实现阶段后更新本文件的阶段进度。

## 已完成内容

- `POST /api/orders` 改为创建 `CREATING_PAYMENT` 订单并投递 `pix-generation` 队列。
- 新增独立 `start:worker` 入口处理 Pix 生成。
- 新增 ChatGPT 代理池、Stripe 代理池、代理失败冷却和维护模式。
- 客户追踪页支持 `CREATING_PAYMENT` 队列位置、前方数量、生成中数量和预计等待。
- Dashboard 增加队列等待、处理中、失败任务、最老等待、平均生成耗时、成功率和代理健康。
- 工人页继续只显示 `PENDING_PAYMENT`，前端对错误推送做状态过滤。
