## 关键决策

- 前端直接改成简体中文，不引入 i18n 框架。
- 支付生成前失败不消耗兑换码；生成 Pix 并成功写入 `PENDING_PAYMENT` 后，兑换码才视为已消费。
- 管理员取消 `CREATING_PAYMENT` 后，外部支付结果不能把订单改回待支付；该场景返回 `ORDER_STATE_CHANGED`。
- `CREATING_PAYMENT` 只作为客户侧处理中状态，不进入工人待处理列表。
- Recharts 2.x 维护风险本轮只登记，不升级。
- 非 0 应付金额代表账号无当前优惠资格，不尝试继续支付；只有应付金额为 0 才继续创建 Pix 并确认 `expected_amount=0`。
- 本轮只保存一个全局代理，不做多代理池；重试只针对同一个代理的短暂网络抖动。
- 为使用审计干净的 `undici` ProxyAgent，项目 Node.js 要求同步提升为 `>=20.18.1`。
- 数据库事务只用于极短原子写入；ChatGPT/Stripe 外部请求绝不放进 Prisma transaction。

## 必需笔记

- 本轮会提交 Prisma migration，因为订单状态枚举需要增加 `CREATING_PAYMENT`。
- 追加迁移只处理强匹配旧失败数据：`FAILED` 且没有 `checkoutSessionId/paymentMethodId/pixCode` 的订单。
- `ORDER_STATE_CHANGED` 使用 HTTP 409，前端展示“订单状态已变化，请重新提交或联系管理员”。
- 外部错误只允许写入安全中文文案，原始错误只能进服务端日志。
- 复制 `.env.example` 后必须替换所有 `REPLACE_WITH_*` 值，否则服务端会拒绝启动。
- 代理设置响应只返回脱敏代理，不返回代理密码；日志只记录代理是否启用、目标域名、重试次数和状态码。
- 新增 `system_settings` Prisma migration，当前用于保存全局 HTTP 代理配置。
- `ORDER_CREATE_BUSY` 使用 HTTP 409，表示数据库短时繁忙或 Prisma 事务关闭类问题，客户可重试。
